# 功能规划文档

## 宪法检查

在开始规划前，请确认以下宪法原则：

- ✅ **技术栈**: JDK 21, Spring Boot 3.5.9, Spring Cloud 2025.0.1, Spring Cloud Alibaba 2025.0.0.0
- ✅ **API 设计**: 遵循 RESTful 风格
- ✅ **代码注释**: 使用中文注释
- ✅ **代码复用**: 提取公共方法，避免重复代码
- ✅ **模块化**: 公共配置归属 `atlas-common-infra-web`

## 功能概述

将异常处理相关的自动配置入口迁移到公共模块，确保业务服务只需依赖公共模块即可自动启用统一异常响应能力，减少显式导入配置的重复成本并降低启动风险。

## 技术方案

### 架构设计

- `atlas-common-infra-web` 提供自动配置入口，负责注册统一异常处理器相关组件
- 业务服务不再显式 `@Import` 配置类，依赖引入后自动生效
- 当公共模块不可用或自动配置加载失败时，服务回退到默认异常处理机制

### 技术选型

- Spring Boot 自动配置机制（基于官方自动配置入口）
- 使用 Spring 提供的条件加载能力确保依赖缺失时可降级

## 实施计划

### 阶段 1: 自动配置入口迁移

- 在 `atlas-common-infra-web` 中新增自动配置入口并注册异常处理相关组件
- 确保配置类具备条件加载与降级策略
- 在 `atlas-system` 中移除显式导入配置的代码

### 阶段 2: 验证与回归

- 验证业务服务启动（公共模块可用/不可用两种场景）
- 验证统一响应结构保持不变
- 回归现有接口与异常处理链路

## 风险评估

- 自动配置未生效导致统一响应失效：通过条件加载与启动日志提示降低排查成本
- 服务启动时序问题：通过自动配置标准入口与条件判断规避

## 验收标准

- 业务服务移除显式配置后仍能自动启用统一异常响应
- 成功/失败响应结构保持一致
- 公共模块不可用时服务仍能启动并回退到默认异常响应
