# 功能规格说明：微服务 Dockerfile 创建

## 宪法检查

本规格说明必须符合项目宪法要求：

- ✅ **中文注释**: 所有配置文件使用中文注释
- ✅ **模块化**: 功能归属正确的模块
- ✅ **Dockerfile**: 可启动微服务需创建 Dockerfile.build 和 Dockerfile.run

## 功能概述

### 目标

为 Atlas 项目中所有可独立启动的微服务创建标准化的容器化配置，确保每个服务都能够独立构建和运行在容器环境中。

### 业务价值

1. **环境一致性**: 开发、测试、生产环境保持一致，减少"在我机器上能运行"的问题
2. **部署标准化**: 统一的容器化方案便于运维团队管理和部署
3. **可扩展性**: 容器化服务便于水平扩展，支持云原生部署架构
4. **隔离性**: 每个服务独立运行在容器中，互不影响

## 涉及的微服务

根据项目宪法原则 17，以下可独立启动的微服务需要创建 Dockerfile：

| 服务名称 | 模块路径 | 服务描述 |
|----------|----------|----------|
| atlas-gateway | `/atlas-gateway` | API 网关服务，负责路由转发、限流熔断 |
| atlas-auth | `/atlas-auth` | 认证授权服务，负责用户认证和 Token 管理 |
| atlas-system | `/atlas-service/atlas-system` | 系统管理服务，负责用户、角色、权限管理 |

## 功能需求

### FR-1: 编译阶段配置

每个微服务必须提供编译阶段的容器配置，满足以下需求：

- **FR-1.1**: 支持完整的项目构建流程
- **FR-1.2**: 使用项目指定的 JDK 21 版本
- **FR-1.3**: 包含构建工具环境（Maven）
- **FR-1.4**: 输出可执行的应用程序包

### FR-2: 运行阶段配置

每个微服务必须提供运行阶段的容器配置，满足以下需求：

- **FR-2.1**: 使用轻量级的运行时环境
- **FR-2.2**: 使用项目指定的 JDK 21 运行时
- **FR-2.3**: 配置合理的资源参数
- **FR-2.4**: 暴露服务对应的端口
- **FR-2.5**: 提供健康检查配置

### FR-3: 配置标准化

所有微服务的容器配置必须遵循统一标准：

- **FR-3.1**: 目录结构一致（`docker/` 目录）
- **FR-3.2**: 文件命名规范一致（`Dockerfile.build`、`Dockerfile.run`）
- **FR-3.3**: 配置注释使用中文
- **FR-3.4**: 遵循最佳实践（镜像层优化、安全配置）

## 服务端口规划

| 服务名称 | 默认端口 | 用途说明 |
|----------|----------|----------|
| atlas-gateway | 8080 | API 网关入口 |
| atlas-auth | 8081 | 认证授权服务 |
| atlas-system | 8082 | 系统管理服务 |

## 用户场景与测试

### 场景 1: 开发人员本地构建

**角色**: 开发人员
**目标**: 在本地环境中构建微服务镜像

**流程**:
1. 开发人员进入微服务模块目录
2. 使用编译阶段配置构建应用
3. 构建完成后获得可执行的应用程序包
4. 使用运行阶段配置创建运行镜像

**验收标准**:
- 构建过程无错误
- 生成的应用程序包可正常执行

### 场景 2: 运维人员部署服务

**角色**: 运维人员
**目标**: 在生产环境中部署微服务

**流程**:
1. 运维人员获取构建好的运行镜像
2. 配置必要的环境变量
3. 启动容器并验证服务状态
4. 通过健康检查确认服务可用

**验收标准**:
- 服务在容器中正常启动
- 健康检查通过
- 服务端口可访问

### 场景 3: CI/CD 自动化构建

**角色**: CI/CD 系统
**目标**: 自动化构建和部署微服务

**流程**:
1. 代码提交触发 CI/CD 流水线
2. 使用编译阶段配置自动构建应用
3. 使用运行阶段配置创建运行镜像
4. 推送镜像到镜像仓库

**验收标准**:
- 构建过程可自动化执行
- 构建结果可复现
- 镜像可正常推送到仓库

## 成功标准

| 标准编号 | 描述 | 衡量指标 |
|----------|------|----------|
| SC-1 | 所有可启动微服务都有完整的容器配置 | 3 个微服务 × 2 个配置文件 = 6 个文件 |
| SC-2 | 容器配置目录结构符合宪法规范 | 所有服务的 `docker/` 目录结构一致 |
| SC-3 | 服务可在容器中正常启动 | 容器启动后 30 秒内通过健康检查 |
| SC-4 | 配置文件包含中文注释 | 每个配置文件的关键配置项都有中文说明 |
| SC-5 | 运行镜像体积合理 | 单个运行镜像体积不超过 500MB |

## 非功能需求

### 安全性

- 运行时容器使用非 root 用户
- 不在镜像中包含敏感信息（密码、密钥等）
- 使用官方或可信的基础镜像

### 可维护性

- 配置文件结构清晰，易于理解
- 关键配置项使用中文注释说明
- 版本信息可追溯

### 性能

- 编译镜像利用构建缓存提高构建速度
- 运行镜像保持精简，减少启动时间

## 假设与约束

### 假设

1. 项目使用 Maven 作为构建工具
2. 所有微服务使用 Spring Boot 打包为可执行 JAR
3. 容器运行环境支持 Docker 或兼容的容器运行时

### 约束

1. 必须使用 JDK 21 版本（宪法原则 1）
2. 必须遵循宪法原则 17 规定的目录结构和命名规范
3. 配置注释必须使用中文（宪法原则 7）

## 范围边界

### 包含

- 为 atlas-gateway、atlas-auth、atlas-system 创建 Dockerfile
- 编译阶段和运行阶段的 Dockerfile 配置
- 配置文件的中文注释

### 不包含

- CI/CD 流水线配置（仅提供可用于 CI/CD 的 Dockerfile）
- 容器编排配置（如 docker-compose、Kubernetes 配置）
- 镜像仓库的配置和管理
- 公共模块（atlas-common-*）和 API 定义模块（atlas-service-api）的 Dockerfile
